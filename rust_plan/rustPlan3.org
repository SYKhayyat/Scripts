** TODO Day 97: Tokio Setup

*** Reading

    Tokio documentation (https://tokio.rs)
    Async book: Getting Started (https://rust-lang.github.io/async-book/01_getting_started/01_chapter.html)
    Tokio tutorial (https://tokio.rs/tokio/tutorial)

*** Summary

Tokio is Rust's most popular async runtime. It provides the executor that runs async code, along with async I/O, timers, channels, and synchronization primitives. Without a runtime, async functions cannot execute.

Async functions in Rust return futures—values representing computations that will complete later. The await keyword suspends execution until a future completes, allowing other tasks to run meanwhile. This cooperative multitasking is more efficient than thread-per-task for I/O-bound work.

Adding Tokio requires specifying features. The full feature set includes everything: runtime, I/O, timers, channels, and macros. For production, you might enable only what you need, but full is fine for development.

The tokio::main attribute macro transforms your main function into an async context. It creates the runtime and blocks on your async main. Without this, you cannot use await in main.

Inside async main, you can call async functions and await their results. The runtime handles scheduling, running your code when futures are ready and parking them when waiting.

Understanding the difference between sync and async is crucial. Sync code runs to completion, blocking the thread. Async code can pause at await points, letting other async code run. This concurrency enables parallelism for I/O operations.

For Nexus, async enables querying all backends simultaneously rather than one at a time. It enables parallel downloads. It keeps the UI responsive during long operations.

** TODO Day 98: Async/Await Basics

*** Reading

    Async book: Why Async? (https://rust-lang.github.io/async-book/01_getting_started/02_why_async.html)
    Async book: async/.await Primer (https://rust-lang.github.io/async-book/03_async_await/01_chapter.html)
    Tokio tutorial: Hello Tokio (https://tokio.rs/tokio/tutorial/hello-tokio)

*** Summary

Async functions are declared with the async keyword before fn. They can use await inside their body. The return type is implicitly wrapped in a future—an async fn returning String actually returns impl Future<Output = String>.

The await keyword is the core of async programming. It suspends the current task until the awaited future completes. During suspension, the runtime can execute other tasks. When the future completes, execution resumes after the await.

Futures are lazy. Creating a future does not start its execution. Only when you await it or spawn it does execution begin. This differs from threads, which start immediately.

Multiple awaits in sequence execute one after another. Each completes before the next begins. This is still useful because the thread is not blocked—other tasks can run during each await.

Error handling in async functions uses the same Result and question mark patterns. Async functions can return Result, and you can use question mark on async operations that return Result.

Async blocks create futures inline without defining a separate function. The syntax async { ... } creates a future containing the block's code. This is useful for short async operations.

For Nexus, async functions will wrap backend operations. Querying installed packages becomes an async operation. Installing packages becomes async. This foundation enables parallel execution.

** TODO Day 99: Concurrent Futures

*** Reading

    Tokio tutorial: Spawning (https://tokio.rs/tokio/tutorial/spawning)
    tokio::join! macro documentation
    tokio::select! macro documentation

*** Summary

Running futures sequentially does not leverage async's power. To run multiple operations concurrently, you need to execute futures in parallel. Tokio provides several mechanisms.

The join! macro runs multiple futures concurrently, waiting for all to complete. It takes multiple futures and returns a tuple of their results. All futures run simultaneously, and join completes when the slowest finishes.

The select! macro runs multiple futures concurrently but returns when the first completes. It cancels the others. This is useful for timeouts or racing alternatives.

Spawning with tokio::spawn creates a new task that runs independently. The spawn function returns a JoinHandle you can await later. Spawned tasks run truly in parallel, potentially on different threads.

The difference between join and spawn matters. Join runs futures on the current task, interleaving execution. Spawn creates separate tasks that can run on separate threads. Join is simpler; spawn is more parallel.

For CPU-bound work, spawn_blocking runs synchronous code on a dedicated thread pool, preventing it from blocking async tasks.

For Nexus, join is appropriate for querying multiple backends—they are I/O bound and benefit from interleaving. Spawn might be used for independent operations that should not block each other.

    Understanding why async traits need special handling

*** Summary

Rust's trait system and async do not natively work together well. Async functions return impl Future, but traits cannot have methods returning impl Trait due to language limitations. The async-trait crate solves this.

The async_trait attribute macro transforms async trait methods into methods returning boxed futures. This adds a small allocation overhead but enables async methods in traits.

Applying async_trait requires adding it to both the trait definition and each impl block. The macro handles the transformation transparently.

Your PackageManager trait methods need to become async. Listing installed packages involves I/O. Installing packages involves I/O. These should not block the async runtime.

Return types remain the same conceptually. An async method returning Result<Vec<Package>, Error> still returns that—the async-trait macro handles the future wrapping.

The overhead of async-trait is minimal for I/O-bound operations. The heap allocation for the boxed future is negligible compared to network or disk I/O time.

For Nexus, update your PackageManager trait to use async methods. This is a significant refactor but enables parallel backend operations.

** TODO Day 101: Spawning Tasks

*** Reading

    Tokio tutorial: Shared State (https://tokio.rs/tokio/tutorial/shared-state)
    Understanding task spawning and join handles

*** Summary

Spawning creates independent tasks that run concurrently with your main task. Unlike join which interleaves on one task, spawn distributes work across the runtime's thread pool.

The tokio::spawn function takes a future and returns a JoinHandle. The future begins executing immediately on a runtime thread. The JoinHandle lets you await the result later.

Spawned tasks must be 'static—they cannot borrow non-static data. This is because spawned tasks might outlive the code that spawned them. Use owned data or Arc for sharing.

Collecting results from spawned tasks involves awaiting the JoinHandles. The handle's await returns Result<T, JoinError> where T is the task's output. JoinError indicates the task panicked.

For parallel backend queries, spawn a task for each backend, collect the handles, then await all handles. This runs backends truly in parallel if the runtime has multiple threads.

Error handling must account for both the task result and potential panics. Wrap task code in appropriate error handling and handle JoinError when awaiting.

For Nexus, spawning enables maximum parallelism when querying backends. Each backend runs independently, using all available CPU cores.

** TODO Day 102: Semaphores for Concurrency Limiting

*** Reading

    tokio::sync::Semaphore documentation
    Understanding concurrency control

*** Summary

Unlimited concurrency can overwhelm resources. Running a hundred downloads simultaneously might exhaust network connections. Semaphores limit how many operations run concurrently.

A semaphore holds a number of permits. Acquiring a permit blocks if none are available. Releasing a permit lets another waiter proceed. This controls the concurrency level.

Tokio's Semaphore is async. The acquire method returns a future that completes when a permit is obtained. The permit is automatically released when dropped.

Setting the right limit depends on the resource. Network operations might allow ten concurrent connections. Disk operations might allow fewer to avoid thrashing. Experiment to find good values.

Combining semaphores with spawn creates bounded parallelism. Spawn tasks that first acquire a permit, do work, then release the permit implicitly by dropping.

For Nexus, semaphores can limit concurrent downloads or concurrent backend operations. This prevents overwhelming the system while still enabling parallelism.
