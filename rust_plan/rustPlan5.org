    Appendix C: Ownership Quick Reference

** The Three Rules of Ownership

    Each value in Rust has exactly one owner (a variable)
    There can only be one owner at a time
    When the owner goes out of scope, the value is dropped

** Borrowing Rules

    You can have either:
        One mutable reference (&mut T), OR
        Any number of immutable references (&T)
    References must always be valid (no dangling references)

** Reference Syntax
Syntax	Name	Meaning
T	Owned	Function takes ownership
&T	Shared reference	Function borrows immutably
&mut T	Mutable reference	Function borrows mutably
&'a T	Reference with lifetime	Reference valid for lifetime 'a

** When to Use What
Situation	Use	Reason
Reading data without storing	&T	No need to own
Modifying data in place	&mut T	Need to modify
Storing data in struct	T	Need ownership
Returning created data	T	Transfer ownership
Returning view of input	&T	Borrow from input
Large data, avoid copy	&T	References are cheap
Need independent copy	.clone()	Explicit duplication

** Common Patterns

#+BEGIN_EXAMPLE
Reading function - borrows input

fn process(data: &[Package]) -> usize
Modifying function - borrows mutably

fn update(data: &mut Vec<Package>)
Transforming function - takes and returns ownership

fn transform(data: Vec<Package>) -> Vec<Package>
Parsing function - borrows input, returns owned

fn parse(input: &str) -> Result<Config, Error>
Constructor - takes ownership of parameters

fn new(name: String, version: String) -> Self
Constructor - borrows and clones

fn from_strs(name: &str, version: &str) -> Self
Method reading self

fn display(&self) -> String
Method modifying self

fn mark_installed(&mut self)
Method consuming self (builder pattern)

fn with_version(self, v: &str) -> Self
#+END_EXAMPLE

** Copy vs Move Types
Copy Types (Stack)	Move Types (Heap)
i8, i16, i32, i64, i128	String
u8, u16, u32, u64, u128	Vec<T>
f32, f64	HashMap<K, V>
bool	Box<T>
char	Most structs
&T (references)	PathBuf
Tuples of Copy types	Rc<T>, Arc<T>
Arrays of Copy types	

** String Types
Type	Ownership	Use Case
String	Owned	Storing, modifying strings
&str	Borrowed	Reading strings, literals
&String	Borrowed	Less flexible than &str

Prefer &str in function parameters for flexibility:
#+BEGIN_EXAMPLE
Works with both String and &str

fn process(name: &str)
Only works with String

fn process(name: &String) # Avoid
#+END_EXAMPLE

** Lifetime Annotations

Lifetimes are usually inferred. Explicit annotation needed when:

    Returning references
    Storing references in structs
    Multiple references with different lifetimes

#+BEGIN_EXAMPLE
Reference in struct needs lifetime

struct Parser<'a> {
input: &'a str,
}
Return reference with same lifetime as input

fn first_word<'a>(s: &'a str) -> &'a str
Multiple inputs, specify which lifetime applies to output

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str
#+END_EXAMPLE

** Common Ownership Errors
Error	Meaning	Fix
E0382	Value used after move	Clone, borrow, or restructure
E0502	Mutable borrow while immutably borrowed	Separate borrows in time
E0499	Multiple mutable borrows	Only one &mut at a time
E0505	Move while borrowed	End borrow before move
E0507	Cannot move out of borrowed content	Clone or use reference
E0515	Cannot return reference to local	Return owned value

    Appendix E: Troubleshooting Guide

** Compilation Errors

*** "cannot borrow as mutable because it is also borrowed as immutable"

You have both &T and &mut T references active simultaneously.

Solutions:

    End the immutable borrow before the mutable borrow
    Clone the data to create independent ownership
    Restructure code to avoid simultaneous borrows
    Use interior mutability with RefCell if appropriate

*** "value moved here" / "use of moved value"

Ownership transferred and you tried to use the original.

Solutions:

    Clone before the move if you need both
    Borrow instead of moving
    Restructure to avoid needing the value after transfer

*** "cannot move out of borrowed content"

You tried to take ownership of data you only borrowed.

Solutions:

    Clone the data to get an owned copy
    Work with references instead
    Change the function signature to take ownership

*** "lifetime may not live long enough"

A reference might outlive the data it points to.

Solutions:

    Ensure data lives long enough
    Clone to create owned data
    Restructure to avoid returning references to local data
    Add explicit lifetime annotations

*** "the trait X is not implemented for Y"

Type does not implement a required trait.

Solutions:

    Add derive macro if the trait is derivable
    Implement the trait manually
    Use a different type that implements the trait
    Check you are using the right type

** Runtime Errors

*** "called unwrap() on a None value"

An Option was None when you expected Some.

Solutions:

    Use match or if let to handle None
    Use unwrap_or or unwrap_or_else for defaults
    Use ? to propagate None
    Debug why None occurred

*** "called unwrap() on an Err value"

A Result was Err when you expected Ok.

Solutions:

    Use match to handle both cases
    Use ? to propagate errors
    Use unwrap_or for defaults
    Debug why the error occurred

*** Command not found when running backend

The backend's command is not installed or not in PATH.

Solutions:

    Install the package manager
    Ensure it is in PATH
    Use absolute paths
    Check is_available before using

** Build Issues

*** Very slow compilation

Solutions:

    Use cargo check instead of cargo build for quick validation
    Enable incremental compilation (on by default)
    Use sccache for caching
    Split into multiple crates
    Reduce dependencies

*** Large binary size

Solutions:

    Build with --release
    Enable LTO in Cargo.toml
    Enable strip in Cargo.toml
    Set codegen-units = 1
    Set panic = "abort"
    Audit dependencies for bloat

*** Dependency conflicts

Solutions:

    Run cargo update
    Check for compatible version ranges
    Use cargo tree to understand dependencies
    Consider forking problematic dependencies

** Testing Issues

*** Tests interfere with each other

Tests run in parallel and share state.

Solutions:

    Use tempdir for file operations
    Avoid global state
    Use --test-threads=1 to run sequentially
    Ensure each test is independent

*** Async tests do not run

Missing async test runtime.

Solutions:

    Use #[tokio::test] for tokio-based tests
    Ensure tokio test features enabled
    Check test function is async

** Async Issues

*** Future is not Send

Trying to spawn a future holding non-Send data.

Solutions:

    Clone data before spawning
    Use Arc for shared ownership
    Drop non-Send data before await points
    Use spawn_local for single-threaded

*** "executor not running" panic

Trying to use async runtime features outside runtime.

Solutions:

    Ensure #[tokio::main] on main
    Do not block inside async context
    Use block_on at top level only
