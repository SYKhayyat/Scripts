** TODO Day 25: Vec<T> Deep Dive

*** Reading

    The Rust Book, Chapter 8.1: Storing Lists of Values with Vectors (https://doc.rust-lang.org/book/ch08-01-vectors.html)
    Standard library documentation for Vec (https://doc.rust-lang.org/std/vec/struct.Vec.html)

*** Summary

Vec<T> is Rust's growable array, similar to Java's ArrayList. It stores elements of a single type contiguously in heap memory, providing fast indexed access and efficient iteration.

Creating vectors has several forms. Vec::new() creates an empty vector. The vec! macro creates a vector with initial elements. Vec::with_capacity() pre-allocates space when you know approximately how many elements to expect, avoiding reallocations as the vector grows.

Adding elements uses push for single elements and extend for multiple elements from an iterator. Removing elements uses pop to remove and return the last element, remove to remove at an index and shift remaining elements, and retain to keep only elements matching a predicate.

Accessing elements has safe and unsafe forms. The get method returns Option, giving None for out-of-bounds indices. Direct indexing with brackets panics on out-of-bounds access. Prefer get when the index might be invalid.

Iteration comes in three flavors reflecting ownership. Borrowing with &vec or vec.iter() yields references, leaving the vector intact. Mutable borrowing with &mut vec or vec.iter_mut() allows modifying elements in place. Consuming with vec or vec.into_iter() takes ownership of each element, emptying the vector.

For Nexus, vectors store package lists, backend lists, error collections, and more. When querying a backend for installed packages, you receive a vector. When collecting packages from config files, you build a vector. Understanding vector operations is essential.

Performance considerations matter. Vectors provide O(1) amortized push and O(1) indexed access. However, inserting or removing from the middle is O(n) due to shifting. Pre-allocating with with_capacity prevents reallocations during growth.

** TODO Day 26: HashMap Basics

*** Reading

    The Rust Book, Chapter 8.3: Storing Keys with Associated Values in Hash Maps (https://doc.rust-lang.org/book/ch08-03-hash-maps.html)
    Standard library documentation for HashMap (https://doc.rust-lang.org/std/collections/struct.HashMap.html)

*** Summary

HashMap<K, V> stores key-value pairs with fast lookup by key, similar to Java's HashMap. Keys must implement the Eq and Hash traits, which you can derive for your own types.

Creating hash maps uses HashMap::new() for empty maps or collecting from iterators of tuples. The insert method adds or replaces entries. Looking up uses get, which returns Option<&V>, or direct indexing, which panics if the key is missing.

The entry API enables efficient conditional operations. Entry returns an Entry enum that is either Occupied or Vacant. The or_insert method inserts a default value if the key is missing and returns a mutable reference to the value. The or_insert_with method computes the default lazily. The and_modify method modifies an existing value.

This entry API avoids double lookups. Without it, checking if a key exists and then inserting requires two hash computations. With entry, you compute once and act on the result.

Iteration over hash maps yields key-value pairs. You can iterate over keys alone with keys(), values alone with values(), or both with iter(). Mutable iteration with iter_mut() allows modifying values but not keys.

For Nexus, hash maps are essential. Mapping package names to package details enables fast lookup. Mapping backend names to backend instances organizes your managers. Mapping section names to package lists structures parsed config files.

Hash maps have no ordering. If you need ordered keys, consider BTreeMap instead, which maintains sorted key order at the cost of slower operations.

** TODO Day 27: HashMap Advanced Patterns

*** Reading

    Standard library collections module overview (https://doc.rust-lang.org/std/collections/index.html)
    Re-read the entry API documentation carefully

*** Summary

Advanced HashMap usage involves nested structures, multiple related maps, and sophisticated entry patterns. These patterns appear frequently in real applications like Nexus.

Nested collections like HashMap<Backend, Vec<Package>> group packages by backend. Each backend key maps to a vector of its packages. Building this structure requires combining entry API with vector operations: get the entry for the backend, insert an empty vector if absent, then push the package.

The or_insert_with method takes a closure, computing the default value only when needed. This is more efficient than or_insert when the default is expensive to create. For inserting empty vectors, or_default works when the value type implements Default.

Multiple related maps sometimes serve different access patterns. You might have packages by name for lookup and packages by backend for processing. Maintaining consistency between related maps requires care—consider whether a single data structure with multiple access methods might be cleaner.

BTreeMap provides an ordered alternative. Keys are sorted, so iteration produces elements in key order. Range queries retrieve subsets of keys. The cost is O(log n) operations instead of O(1), and keys must implement Ord rather than Hash.

For Nexus, grouping packages by backend is essential for sync operations. Each backend processes its own packages independently. Building this grouped structure from a flat package list uses the patterns described here.

Consider memory and performance. Large maps benefit from capacity hints. Frequent lookups with rare modifications favor HashMap. Need for ordering or ranges favors BTreeMap.

** TODO Day 28: HashSet

*** Reading

    Standard library documentation for HashSet (https://doc.rust-lang.org/std/collections/struct.HashSet.html)

*** Summary

HashSet<T> stores unique values with fast membership testing. It is like a HashMap with only keys, no values. Elements must implement Eq and Hash.

Creating sets uses HashSet::new() or collecting from an iterator. The insert method adds elements, returning a boolean indicating whether the element was new. The contains method tests membership.

Set operations compute relationships between sets. Union combines elements from both sets. Intersection finds elements present in both. Difference finds elements in the first set but not the second. Symmetric difference finds elements in either set but not both.

These operations return iterators, so you typically collect them into a new set or iterate directly. They do not modify the original sets.

For Nexus, sets are perfect for package tracking. The set of installed packages comes from backend queries. The set of declared packages comes from configuration. The difference between declared and installed is what needs installing. The difference between installed and declared is unmanaged packages.

This set-based logic is at the heart of sync operations. Rather than nested loops comparing lists, set operations express the intent directly and execute efficiently.

Set operations are O(n) where n is the smaller set's size for most operations. Membership testing is O(1). This makes set-based package diffing fast even with thousands of packages.

** TODO Day 29: Iterators Introduction

*** Reading

    The Rust Book, Chapter 13.2: Processing a Series of Items with Iterators (https://doc.rust-lang.org/book/ch13-02-iterators.html)
    Rust By Example: Iterators (https://doc.rust-lang.org/rust-by-example/trait/iter.html)

*** Summary

Iterators are Rust's abstraction for sequential processing. They are lazy—nothing happens until you consume them. This enables chaining operations efficiently without intermediate collections.

The Iterator trait requires implementing next(), which returns Option<Item>. Some yields the next element, None signals exhaustion. All other iterator methods build on next.

Creating iterators from collections uses iter() for borrowing references, iter_mut() for mutable references, and into_iter() for taking ownership. Ranges like 0..10 are iterators directly.

Transforming adaptors produce new iterators. Map applies a function to each element. Filter keeps elements matching a predicate. Take limits to a count. Skip ignores initial elements. Enumerate pairs elements with indices. Chain concatenates iterators. Zip pairs elements from two iterators.

Consuming adaptors produce values. Collect gathers elements into a collection. Count returns the number of elements. Sum adds numeric elements. Fold reduces to a single value with an accumulator. For_each executes a closure for each element. Any tests if any element matches. All tests if all elements match. Find returns the first matching element.

The power is in chaining. You express complex transformations as a pipeline: start with data, filter, map, filter again, collect. No intermediate collections are allocated—elements flow through the pipeline one at a time.

For Nexus, iterators process package lists, parse file lines, filter by criteria, transform between representations. Mastering iterators makes your code concise and efficient.

** TODO Day 30: Iterator Practice

*** Reading

    The Rust Book, Chapter 13.4: Comparing Performance: Loops vs. Iterators (https://doc.rust-lang.org/book/ch13-04-performance.html)
    Standard library iter module documentation (https://doc.rust-lang.org/std/iter/index.html)

*** Summary

Today deepens iterator mastery with advanced methods and performance understanding. Iterators in Rust are zero-cost abstractions—they compile to code as efficient as hand-written loops.

Flat_map handles nested structures. If each element produces an iterator, flat_map chains them together. This is useful when each backend provides a package list and you want all packages in one stream.

The partition method splits an iterator into two collections based on a predicate. Elements matching go to the first collection, others to the second. Useful for separating successes from failures.

Collecting into Result works specially. An iterator of Results can be collected into a Result of a collection. If any element is Err, the collection is Err. This enables short-circuit processing where one failure fails the whole operation.

Peekable wraps an iterator to allow looking at the next element without consuming it. Useful for parsers that need lookahead.

The by_ref method borrows the iterator, allowing partial consumption. You might take some elements, then continue with the rest.

Performance of iterators matches or exceeds explicit loops. The compiler inlines iterator methods and optimizes the resulting code. Benchmarks consistently show iterators performing as well as manual loops, sometimes better due to optimizer visibility.

For Nexus, these patterns handle real scenarios. Flat_map processes all packages across all backends. Collecting into Result fails fast on parse errors. Partition separates successful operations from failures.

Traits define shared behavior, enabling polymorphism and abstraction. They are how Nexus works with different backends through a common interface.

** TODO Day 31: Traits Concept

*** Reading

    The Rust Book, Chapter 10.2: Traits: Defining Shared Behavior (https://doc.rust-lang.org/book/ch10-02-traits.html)

*** Summary

Traits define functionality that types can implement. They are similar to Java interfaces but more powerful, supporting default implementations and being applicable to any type, not just classes.

Defining a trait lists method signatures that implementors must provide. Methods can take various forms of self for instance methods or omit self for associated functions.

Default implementations provide fallback behavior. Implementors can override defaults or use them. This reduces boilerplate when many implementors share common logic.

Trait bounds constrain generic types. Writing T: SomeTrait means T must implement SomeTrait. This lets generic functions call trait methods on their parameters.

Multiple traits can be required with plus signs: T: TraitA + TraitB. Impl Trait syntax in return position enables returning any type implementing a trait without naming it explicitly.

For Nexus, the PackageManager trait is central. Every backend—apt, pacman, pip, cargo, and fifteen others—implements this trait. The trait defines what a package manager can do: check availability, list installed packages, install packages, remove packages.

With this trait, code that processes backends does not care which specific backend it has. It calls trait methods, and each backend provides its implementation. This is polymorphism without inheritance.

Design the trait carefully. What operations do all backends support? What parameters do they need? What do they return? The trait is a contract that all nineteen backends must fulfill.

** TODO Day 32: Implementing Traits

*** Reading

    The Rust Book, Chapter 10.2: Continue reading about implementing traits
    Rust By Example: Traits (https://doc.rust-lang.org/rust-by-example/trait.html)

*** Summary

Implementing a trait for a type provides that type's specific behavior for the trait's methods. The impl block names both the trait and the type.

Each method from the trait must be implemented unless it has a default. The implementation can access self and any fields of the implementing type. It can also call other methods, both from the trait and from the type itself.

Struct fields support the implementation. A backend manager might have fields for configuration, logging, or caching. Methods access these fields while fulfilling the trait contract.

Multiple types can implement the same trait, each with different behavior. This is the power of traits—common interface, varied implementation.

The orphan rule constrains what you can implement. You can implement your traits for any type, or any trait for your types, but not a foreign trait for a foreign type. This prevents conflicts between crates.

For Nexus, each backend struct implements PackageManager. AptManager implements it by running apt commands. PacmanManager implements it by running pacman commands. The implementations differ, but the interface is uniform.

Start with a few backends, then expand. Apt and pip are good starting points—one system package manager, one language package manager. Get the pattern right with these before implementing all nineteen.

** TODO Day 33: Multiple Implementors

*** Reading

    The Rust Book, Chapter 10.2: "Returning Types that Implement Traits" section
    The Rust Book, Chapter 17.2: Using Trait Objects That Allow for Values of Different Types (https://doc.rust-lang.org/book/ch17-02-trait-objects.html)

*** Summary

With multiple types implementing the same trait, you can write code that works with any of them. Rust provides two mechanisms: static dispatch with generics and dynamic dispatch with trait objects.

Static dispatch with generics specifies trait bounds. A function taking T: PackageManager works with any type implementing the trait. The compiler generates specialized code for each type used, called monomorphization. This is fast but increases binary size if many types are used.

Dynamic dispatch with trait objects uses dyn Trait. A function taking &dyn PackageManager accepts any implementor at runtime. Method calls go through a vtable, like virtual methods in Java. This is slightly slower but more flexible.

Choosing between them depends on your needs. When you know types at compile time and want maximum performance, use generics. When you need to store different types together or choose types at runtime, use trait objects.

For Nexus, storing all backends together requires trait objects. You cannot have a Vec<T> where T varies per element. Instead, use Vec<Box<dyn PackageManager>> where each element can be a different backend type.

The Box is necessary because trait objects have unknown size—different backends have different struct sizes. Box provides heap allocation with a fixed pointer size.

Consider also when you might use generics. A function that processes one backend at a time might use generics for performance, even if the registry storing backends uses trait objects.

** TODO Day 34: Trait Objects and Box

*** Reading

    The Rust Book, Chapter 17.2: Using Trait Objects (https://doc.rust-lang.org/book/ch17-02-trait-objects.html) - complete
    The Rust Book, Chapter 15.1: Using Box<T> to Point to Data on the Heap (https://doc.rust-lang.org/book/ch15-01-box.html)

*** Summary

Trait objects enable runtime polymorphism. When you need to work with values of different types through a common interface without knowing the concrete types, trait objects are the solution.

Object safety constrains which traits can become trait objects. A trait is object-safe if its methods do not return Self (which would have unknown size) and do not have generic type parameters (which would require monomorphization). Your PackageManager trait should be designed for object safety.

Box<T> is a smart pointer that allocates T on the heap. For trait objects, Box<dyn Trait> owns a heap-allocated value implementing the trait. The Box itself has known size—just a pointer—even though the contained value has unknown size.

Creating trait objects wraps concrete types. Box::new(AptManager::new()) creates a Box<AptManager>, which coerces to Box<dyn PackageManager> when needed. You can explicitly write the type or let coercion handle it.

Method calls on trait objects work transparently. You call manager.name() on a Box<dyn PackageManager> just like on a concrete type. The vtable dispatches to the correct implementation.

For Nexus, the backend registry holds Box<dyn PackageManager> for each available backend. At runtime, you iterate over these, calling trait methods without knowing which specific type each is. This decouples the sync logic from specific backend implementations.

Consider ownership versus borrowing. Box owns its content. For borrowed trait objects, use &dyn Trait or &mut dyn Trait. References work when you do not need ownership, like passing to a function.

** TODO Day 35: Derive Traits

*** Reading

    The Rust Book, Chapter 10.2: "Derivable Traits" section
    Rust Reference: Derive (https://doc.rust-lang.org/reference/attributes/derive.html)

*** Summary

Derive macros automatically implement traits based on your type's structure. They generate code you would otherwise write by hand, reducing boilerplate and bugs.

Common derivable traits include Debug for {:?} formatting, Clone for .clone(), PartialEq for == comparison, Eq for complete equality, Hash for hash map keys, Default for default values, and PartialOrd/Ord for ordering.

Derive requirements cascade. To derive Clone, all fields must implement Clone. If a field does not, you must implement Clone manually or change the field type.

Not all traits are derivable. Display must be implemented manually because there is no obvious default format. Custom traits you define are not derivable unless you create procedural macros.

Serde provides derive macros for serialization. Deriving Serialize and Deserialize enables converting your types to and from formats like JSON and TOML. This is essential for configuration parsing.

For Nexus, most of your structs should derive Debug for logging and testing. Clone is needed if you copy values. PartialEq enables testing assertions. Serde derives enable configuration loading.

Think about each derive. Does this type need to be cloned? Compared? Hashed? Only derive what you need—unnecessary derives can have compile-time and sometimes runtime costs.

** TODO Day 37: Module Basics

*** Reading

    The Rust Book, Chapter 7.1: Packages and Crates (https://doc.rust-lang.org/book/ch07-01-packages-and-crates.html)
    The Rust Book, Chapter 7.2: Defining Modules to Control Scope and Privacy (https://doc.rust-lang.org/book/ch07-02-defining-modules-to-control-scope-and-privacy.html)

*** Summary

Rust organizes code into modules, crates, and packages. Understanding this hierarchy is essential for structuring Nexus properly.

A package is a Cargo project with a Cargo.toml. It contains one or more crates. A crate is a compilation unit—either a binary (executable) or a library. Nexus is a package containing a binary crate.

Modules organize code within a crate. They create namespaces, control privacy, and group related functionality. You declare modules with the mod keyword.

Module declarations connect to files. Declaring mod backend in main.rs tells Rust to look for backend.rs or backend/mod.rs. The file's contents become the module's contents.

Privacy defaults to private. Items in a module are invisible outside it unless marked pub. This encapsulation hides implementation details, exposing only the intended interface.

The use keyword brings items into scope. Instead of writing crate::backend::Backend everywhere, you use crate::backend::Backend once, then just Backend.

Path prefixes navigate the module tree. crate:: starts at the crate root. super:: goes to the parent module. self:: explicitly refers to the current module.

For Nexus, modules separate concerns. A backend module contains backend-related code. A config module handles configuration. A sync module implements synchronization logic.

*** Summary

Larger modules benefit from directory structure. Instead of one large file, you have a directory with multiple files, each handling part of the module's responsibility.

A directory module uses a mod.rs file as its root. The file src/backends/mod.rs is the root of the backends module. It declares submodules with mod statements.

Submodules live in sibling files. Declaring mod apt in mod.rs looks for apt.rs in the same directory. Each submodule can be further nested if needed.

Re-exporting with pub use simplifies imports for users. If apt.rs defines AptManager, and mod.rs re-exports it with pub use apt::AptManager, users can import directly from backends rather than navigating to backends::apt.

Nested use paths reduce repetition. Instead of separate use statements for each item, combine them: use crate::backends::{AptManager, PacmanManager, PipManager}.

Glob imports with * import everything public from a module. Use sparingly—they make it unclear where items come from. They are appropriate for preludes or test modules.

For Nexus, the backends module is a directory containing each backend manager. The mod.rs declares all backends and re-exports their types. Users import from backends without knowing the internal structure.

** TODO Day 39: Visibility

*** Reading

    The Rust Book, Chapter 7.5: Separating Modules into Different Files (https://doc.rust-lang.org/book/ch07-05-separating-modules-into-different-files.html)
    Rust Reference: Visibility and Privacy (https://doc.rust-lang.org/reference/visibility-and-privacy.html)

*** Summary

Visibility controls who can access items. The default is private—only the defining module can access it. Pub modifiers grant broader access.

Plain pub makes an item public to everyone, including external crates if this is a library. Use this for your intended public API.

The pub(crate) modifier makes an item public within the crate but not to external users. Use this for internal APIs that multiple modules need but external code should not use.

The pub(super) modifier makes an item public to the parent module. This is useful for helpers that only the parent needs.

The pub(in path) modifier restricts visibility to a specific ancestor module. This is rarely needed but provides fine-grained control.

Struct fields have individual visibility. A pub struct can have private fields, accessed only through methods. This enables encapsulation—the struct controls how fields are modified, maintaining invariants.

Enum variants follow the enum's visibility. If the enum is pub, all variants are pub. You cannot have private variants in a public enum.

For Nexus, think about what should be exposed. The public API includes types and functions users call. Internal helpers should be pub(crate) or private. Struct fields might be private with accessor methods.

*** Summary

Unit tests verify individual functions and methods. In Rust, they live in the same file as the code they test, in a tests submodule marked with cfg(test).

The cfg(test) attribute means the module only compiles when testing. This keeps test code out of production builds, reducing binary size and compile time.

Test functions are marked with the test attribute. They typically call the code being tested and make assertions about the results. If assertions pass, the test passes. If any assertion fails or the function panics, the test fails.

Assertion macros include assert! for boolean conditions, assert_eq! for equality, and assert_ne! for inequality. They print helpful messages on failure showing expected versus actual values.

Tests can be marked should_panic if the expected behavior is a panic. The optional expected parameter checks the panic message contains specific text.

The ignore attribute skips a test normally but allows running it explicitly. Use this for slow tests you do not want to run constantly.

Because tests are in the same module as code, they can access private items. This enables testing internal functions without exposing them publicly.

For Nexus, unit tests verify parsing functions, validation logic, data transformations, and more. Each module should have a tests submodule with thorough coverage.
