    Week 9: CLI Parsing with Clap

The command-line interface is how users interact with Nexus. A well-designed CLI makes the tool pleasant to use. Clap is Rust's premier CLI parsing library, providing argument parsing, help generation, and validation.

** TODO Day 49: Clap Basics

*** Reading

    clap documentation (https://docs.rs/clap)
    clap tutorial (https://docs.rs/clap/latest/clap/_tutorial/index.html)
    clap derive reference (https://docs.rs/clap/latest/clap/_derive/index.html)

*** Summary

Clap is a powerful command-line argument parser that handles the tedious parts of CLI development: parsing arguments, generating help text, validating input, and producing error messages. It supports two APIs: builder and derive. The derive API uses Rust attributes to define your CLI declaratively.

The derive approach defines a struct representing your CLI. Fields become arguments or flags. Attributes control how they appear and behave. Clap generates the parsing code at compile time.

Adding clap requires specifying features in your dependency. The derive feature enables the attribute macros. Other features add functionality like shell completions or styling.

The Parser trait marks your CLI struct. Deriving it provides a parse() method that reads command-line arguments and returns a populated struct or exits with an error message.

Clap automatically generates help text from doc comments. A comment above a field becomes that field's help text. A comment above the struct becomes the program description. This keeps documentation with the code.

Version and about information comes from Cargo.toml by default if you add the cargo feature, or you can specify them explicitly with attributes.

For Nexus, the CLI struct holds global flags like dry-run and verbose, plus a subcommand enum for the specific command being run. This structure mirrors how users invoke the tool: nexus --dry-run sync.

*** Summary

Subcommands let users specify different operations. Rather than one monolithic command, you have nexus sync, nexus clean, nexus unmanaged, and so on. Each subcommand can have its own arguments.

The Subcommand derive marks an enum where each variant is a subcommand. Variant names become subcommand names, converted to lowercase and kebab-case by default.

Connecting the subcommand enum to your main Cli struct uses the subcommand attribute on a field. The field type is your Commands enum.

Each variant can hold data representing that subcommand's arguments. A tuple variant holds positional arguments. A struct variant holds named arguments and flags. An empty variant takes no arguments.

Doc comments on variants become subcommand descriptions shown in help. Users see these when running nexus --help to list available commands.

Handling subcommands in main uses pattern matching. Match on the command field to determine which subcommand was invoked, then extract any arguments from the variant.

For Nexus, you need subcommands for sync, clean, unmanaged, orphans, and possibly help. Each has different arguments: sync might have flags for removing untracked packages, clean might specify a backend.

Short flags are single-character alternatives to long flags. Users type -n instead of --dry-run for brevity. Good short flag choices follow conventions and avoid conflicts.

Combined short flags work automatically. Users can type -vyn instead of -v -y -n. Clap handles this without additional configuration.

For Nexus, match the LiNix conventions where possible: -n for dry-run, -y for yes, -v for verbose, -b for backend, -c for config, -g for groups-dir, -r for remove-bloatware.

*** Summary

TOML is a configuration file format designed for humans. It is more readable than JSON or YAML, with clear syntax for sections, key-value pairs, and nested structures.

The toml crate parses TOML into Rust types. Combined with serde, it deserializes TOML directly into your structs. This eliminates manual parsing and provides type safety.

Adding dependencies requires both toml and serde. The serde crate needs the derive feature for automatic serialization implementation.

Basic parsing uses toml::from_str with a type annotation or turbofish. It returns a Result containing either the parsed value or an error with line and column information.

Serde works through the Deserialize trait. Deriving it on your struct tells serde how to populate fields from serialized data. Field names match TOML keys by default.

Error handling for TOML parsing should preserve line information. When presenting parse errors to users, include where in the file the error occurred so they can fix it.

For Nexus, the config.toml file contains enabled backends, hostname groups, and hooks. Start by parsing the enabled_backends section, then expand to handle the full configuration.

*** Summary

Serde derives generate serialization code from struct definitions. The Deserialize derive enables reading data into your types. The Serialize derive enables writing data from your types.

Field attributes customize how individual fields are handled. The rename attribute changes the key name in the serialized format. The default attribute uses Default::default() if the field is missing. The skip attribute excludes a field from serialization.

The alias attribute accepts alternate names. A field with alias can be read from either its primary name or the alias. Useful for backward compatibility or alternate conventions.

The flatten attribute inlines a nested struct's fields into the parent. Useful when configuration is logically grouped but serialized flat.

Container attributes apply to the whole struct or enum. The default attribute on a struct uses Default for all missing fields. The rename_all attribute changes the naming convention for all fields.

Configuration files should be lenient. Users should not need to specify every field, only those they want to change from defaults. Serde's default handling enables this.

Field-level default functions provide custom defaults. When Default::default() is not right, write a function returning the default value and reference it with default = "function_name".

Combining approaches lets you have some fields required, some with defaults, some truly optional. Required fields have neither Option nor default. Defaulted fields have default. Optional fields are Option possibly with default = None explicit.

*** Summary

Commands can hang indefinitely if something goes wrong. Network issues during downloads, broken package scripts, or waiting for input can all cause hangs. Timeouts prevent infinite waits.

Implementing timeout requires running the command asynchronously and waiting with a limit. The wait-timeout crate provides this functionality. Alternatively, spawn the command and implement timeout manually with threads.

Configurable timeouts let users adjust for their situation. Slow networks might need longer timeouts. Fast systems might want shorter for faster failure detection.
