*** Summary

Rust is a systems programming language that runs blazingly fast, prevents segfaults, and guarantees thread safety. Unlike Java, which compiles to bytecode and runs on the JVM, Rust compiles directly to native machine code. This means no runtime overhead and no garbage collector.

For Nexus, this matters tremendously. Package managers need to be fast—users run them constantly. The Go-based LiNix is already fast, but Rust can match or exceed that performance while giving us stronger safety guarantees.

You will install three core tools today. First is rustup, the toolchain manager that handles Rust versions and components—think of it like SDKMAN for Java. Your project structure will differ from Java. Instead of src/main/java with package directories, Rust uses a flat src directory with modules defined by files. The Cargo.toml file replaces pom.xml, defining your project metadata and dependencies in a clean, readable format called TOML.

The entry point for a Rust program is the main function, written simply as fn main() without the ceremony of public static void main(String[] args). Rust infers that main takes no arguments by default; command-line arguments are accessed differently.

** TODO Day 3: Variables, Mutability, and Basic Types

*** Reading

    The Rust Book, Chapter 3.1: Variables and Mutability (https://doc.rust-lang.org/book/ch03-01-variables-and-mutability.html)
    Rust By Example: Variable Bindings (https://doc.rust-lang.org/rust-by-example/variable_bindings.html)

*** Summary

Here is your first major mental shift from Java: in Rust, variables are immutable by default. When you declare a variable with let, you cannot change its value. This is the opposite of Java, where variables are mutable unless marked final.

This design choice is intentional and powerful. Immutability prevents accidental changes to data, makes code easier to reason about, and enables safe concurrent programming. For Nexus, when we process packages across multiple threads, immutability helps prevent data races without explicit locking.

To make a variable mutable, you must explicitly declare it with let mut. This signals to anyone reading your code that this variable will change. The compiler tracks mutability and ensures you only modify variables declared as mutable.

Shadowing is a Rust concept that may seem strange at first. You can declare a new variable with the same name as an existing one, and the new variable "shadows" the old one. This is not mutation—you are creating a new variable. Shadowing is useful for transforming values while keeping meaningful names.

Constants differ from immutable variables. Constants are declared with const, must have explicit type annotations, and must be set to values computable at compile time. Use constants for values that never change throughout your program, like maximum retry counts or default paths.

For Nexus, think about what data should be immutable. A package's name probably should not change after parsing. The configuration loaded from disk might be immutable once loaded. The list of installed packages might be immutable during a single sync operation, rebuilt rather than mutated.

** TODO Day 4: Primitive Types Deep Dive

*** Reading

    The Rust Book, Chapter 3.2: Data Types (https://doc.rust-lang.org/book/ch03-02-data-types.html)
    Rust Reference: Types chapter, skim for awareness (https://doc.rust-lang.org/reference/types.html)

*** Summary

Rust has a rich set of primitive types, and choosing the right one matters for both correctness and performance. Unlike Java where int is always 32 bits, Rust lets you choose exactly the size you need.

Integer types come in signed and unsigned variants, from 8 bits to 128 bits. The naming is logical: i32 is a signed 32-bit integer, u64 is an unsigned 64-bit integer. For indexing into collections, use usize, which matches your system's pointer size—64 bits on modern machines.

For Nexus, type choices matter. Package download sizes should be u64 since they cannot be negative and might exceed 4GB. Exit codes from processes might be i32. Counts of packages should probably be usize since you will use them for indexing.

Floating point numbers come as f32 and f64, with f64 being the default. You probably will not need these much in Nexus except perhaps for progress percentages.

Strings in Rust are more complex than in Java, and understanding this early prevents confusion later. There are two main string types: String is an owned, heap-allocated, growable string, while str (usually seen as &str) is a borrowed reference to string data. Think of String as owning its data like Java's String, while &str is more like a view into string data that lives elsewhere.

String literals like "hello" are &str, not String. They point to data baked into your compiled binary. To get an owned String, you convert with methods like to_string() or String::from(). This distinction relates to ownership, which we cover next week.

Tuples group multiple values of potentially different types with a fixed size. Arrays hold multiple values of the same type with a fixed size. For dynamic sizing, you will use Vec, covered in Phase 1.

** TODO Day 5: Functions

*** Reading

    The Rust Book, Chapter 3.3: Functions (https://doc.rust-lang.org/book/ch03-03-how-functions-work.html)
    Rust By Example: Functions (https://doc.rust-lang.org/rust-by-example/fn.html)

*** Summary

Functions in Rust will feel somewhat familiar from Java, but with important differences in syntax and behavior. The most significant is that Rust functions use snake_case naming rather than camelCase.

Function signatures declare parameter types and return types explicitly. The return type comes after an arrow. If a function returns nothing meaningful, it returns the unit type, which is implicit and can be omitted from the signature.

The profound difference from Java is that Rust functions are expression-based. The last expression in a function, without a semicolon, becomes the return value. This is idiomatic Rust—you will rarely see explicit return statements except for early returns.

Understanding statements versus expressions is crucial. A statement performs an action and returns nothing—it ends with a semicolon. An expression evaluates to a value—it has no semicolon. A block of code enclosed in braces is itself an expression, evaluating to its last expression.

This means you can write concise functions where the entire body is an expression. You can also assign the result of an if-else or a block to a variable. This expressiveness makes Rust code compact yet readable.

For Nexus, functions will form the backbone of your logic: functions to parse configuration, functions to query backends, functions to calculate package differences, functions to execute installations. Good function design with clear inputs and outputs makes code testable and maintainable.

Documentation comments with three slashes become part of generated documentation. Get in the habit of documenting every public function with its purpose, parameters, return value, and examples.

** TODO Day 6: Control Flow

*** Reading

    The Rust Book, Chapter 3.5: Control Flow (https://doc.rust-lang.org/book/ch03-05-control-flow.html)
    Rust By Example: Flow of Control (https://doc.rust-lang.org/rust-by-example/flow_control.html)

*** Summary

Control flow in Rust has familiar constructs with powerful twists. The most important: if is an expression, not just a statement. This means if-else can return a value, eliminating the need for ternary operators.

When using if as an expression, both branches must return the same type. The compiler enforces this, preventing subtle bugs where different branches return incompatible types.

Rust provides three loop constructs. The loop keyword creates an infinite loop—use break to exit. The while loop continues while a condition is true. The for loop iterates over an iterator, most commonly a range or collection.

The loop construct has a unique feature: it can return a value via break. This is useful when searching for something—loop until found, then break with the found value.

Ranges are expressed with dot-dot notation: 0..10 produces 0 through 9 (exclusive end), while 0..=10 produces 0 through 10 (inclusive end). This is common for iterating a specific number of times.

Loop labels allow breaking from nested loops. You can label a loop with a tick-prefixed name like 'outer, then break 'outer to exit that specific loop from anywhere inside it. For Nexus, this might be useful when searching across multiple backends for a package.

For iteration with an index, the enumerate method on iterators produces tuples of index and value. This is cleaner than manually tracking an index.

    Week 2: Ownership - The Heart of Rust

This is the most important week. Ownership is what makes Rust unique and what enables its safety guarantees without garbage collection. Take your time here—deep understanding prevents frustration later.

** TODO Day 7: Understanding Ownership

*** Reading

    The Rust Book, Chapter 4.1: What is Ownership? (https://doc.rust-lang.org/book/ch04-01-what-is-ownership.html)
    Read this chapter twice. It is that important.

*** Summary

Ownership is Rust's approach to memory management. Every value in Rust has exactly one owner—a variable that is responsible for that value. When the owner goes out of scope, the value is automatically cleaned up. No garbage collector needed, no manual memory management required.

This system prevents entire categories of bugs that plague other languages: use-after-free, double-free, dangling pointers, data races. The compiler checks ownership rules at compile time, so these bugs are caught before your code ever runs.

The key insight is move semantics. When you assign one variable to another for heap-allocated types like String, ownership moves. The original variable becomes invalid. This prevents two variables from thinking they own the same data, which would cause double-free when both go out of scope.

Stack-allocated types like integers and booleans implement the Copy trait. For these, assignment copies the bits rather than moving ownership. Both variables remain valid because the copy is cheap and safe.

Understanding the difference between stack and heap allocation helps. Stack allocation is fast but size must be known at compile time. Heap allocation is flexible but requires tracking who is responsible for freeing the memory. Rust's ownership tracks that responsibility.

For Nexus, ownership affects how you design data flow. When a function parses configuration, does it return owned data or references? When you pass a package list to a function, does that function take ownership or borrow? These decisions shape your API.

The clone method creates a deep copy of heap data when you genuinely need two owned copies. Cloning is explicit in Rust, unlike Java where object assignment might invisibly share or copy depending on the type. Explicit cloning makes the cost visible.

** TODO Day 8: Borrowing with &

*** Reading

    The Rust Book, Chapter 4.2: References and Borrowing (https://doc.rust-lang.org/book/ch04-02-references-and-borrowing.html) — first half, up to "Mutable References"

*** Summary

Ownership solves memory safety but creates a usability problem. If functions always take ownership, you lose access to your data after passing it. You would need to clone everything or have every function return the data it received.

Borrowing is the solution. A reference, written with an ampersand, lets you refer to a value without taking ownership. The original owner keeps ownership while the reference exists.

Think of it like lending a book. You still own the book, but someone else can read it. They must return it (the reference goes out of scope), and while they have it, certain constraints apply.

Immutable references, written as &T, allow reading but not modifying. You can have multiple immutable references to the same data simultaneously—many readers are safe.

Function parameters that borrow use reference types. A function taking &String borrows a string without taking ownership. The caller can keep using their string after the function returns.

An important idiom: prefer &str over &String in function parameters. The type &str is more flexible—it accepts both String values (borrowed as a slice) and string literals. This makes your functions work with more kinds of input.

References must always be valid. Rust prevents dangling references at compile time. You cannot return a reference to data that will be dropped when the function ends.

For Nexus, most functions that read data should borrow it. Functions that list packages, format output, check installation status—all of these can borrow rather than own. This avoids unnecessary cloning and makes APIs cleaner.

** TODO Day 9: Mutable Borrowing with &mut

*** Reading

    The Rust Book, Chapter 4.2: References and Borrowing (https://doc.rust-lang.org/book/ch04-02-references-and-borrowing.html) — second half, "Mutable References" onwards

*** Summary

Sometimes you need to modify borrowed data. Mutable references, written as &mut T, allow this. But Rust enforces a crucial rule: you can have either one mutable reference or any number of immutable references, but not both.

This rule prevents data races at compile time. A data race occurs when two pointers access the same data, at least one writes, and access is unsynchronized. By forbidding simultaneous mutable and immutable references, Rust guarantees that readers see consistent data and writers have exclusive access.

For Nexus running parallel operations, this rule is your friend. The compiler catches potential races before they become hard-to-debug runtime bugs.

To create a mutable reference, the original variable must be declared with mut. Then you create the reference with &mut. Functions accepting mutable references can modify the caller's data in place.

Non-lexical lifetimes make the borrow checker smarter than it used to be. A reference's lifetime ends at its last use, not at the end of its scope. This means you can start a mutable borrow after you are done using immutable borrows, even within the same function.

Mutable references are exclusive but temporary. Once the mutable reference goes out of scope or is last used, you can create new references. The borrow checker tracks this precisely.

Consider when Nexus needs mutable access. Adding packages to a list requires mutable access to the list. Updating a configuration value requires mutable access. But reading package information, formatting output, checking conditions—these need only immutable access.

** TODO Day 10: Slices

*** Reading

    The Rust Book, Chapter 4.3: The Slice Type (https://doc.rust-lang.org/book/ch04-03-slices.html)

*** Summary

A slice is a reference to a contiguous portion of a collection. Rather than referencing the whole thing, a slice references a range. Slices are views into data—they do not own what they point to.

String slices, written as &str, are views into string data. When you have a String and want to pass just part of it to a function, you create a slice. String literals are themselves &str slices pointing to data in your compiled binary.

The range syntax specifies which portion: [0..5] means indices 0 through 4, [..5] means from the start through 4, [2..] means from index 2 to the end, [..] means the entire thing. Adding an equals sign makes the end inclusive: [0..=5] means 0 through 5.

Array and vector slices work similarly, written as &[T]. A function taking &[String] can accept slices of both arrays and vectors, making it flexible.

Slices are powerful for performance. When parsing a configuration file, you can work with slices of the file content without copying strings. When processing packages, you can pass slices of a package list to functions without cloning the list.

For Nexus, slices are essential for efficient parsing. The config file is loaded once as a String, then slicing extracts sections, keys, and values without additional allocation. The package list from a backend is parsed by slicing output lines.

Safe slice indexing with the get method returns Option—Some with the element or None if the index is out of bounds. Direct indexing with brackets panics on out-of-bounds access. Prefer get when the index might be invalid.

** TODO Day 11: Ownership in Practice

*** Reading

    Review the summaries from Days 7-10
    Rust By Example: Ownership and moves (https://doc.rust-lang.org/rust-by-example/scope/move.html)
    Rust By Example: Borrowing (https://doc.rust-lang.org/rust-by-example/scope/borrow.html)

*** Summary

Today we synthesize ownership concepts and apply them to designing Nexus. The goal is developing intuition for ownership patterns so you can make these decisions automatically.

The core question when writing a function is: what should this function do with its inputs? If it needs to store them or return ownership, take owned values. If it only needs to read them, borrow immutably. If it needs to modify them in place, borrow mutably.

For return values: if you create data inside a function, you typically return it owned, transferring ownership to the caller. If you receive borrowed data and want to return a view into it, you return a reference with the same lifetime.

Common patterns emerge. Parsing functions typically borrow input (the string to parse) and return owned output (the parsed structure). Query functions borrow the data structure and return owned results or borrowed references. Modification functions take mutable borrows and change data in place.

Cloning has its place. When you genuinely need independent copies, clone explicitly. But unnecessary cloning hurts performance. The Rust community emphasizes "zero-cost abstractions"—pay only for what you use. Cloning everything contradicts this principle.

For Nexus specifically, consider the data flows. Configuration is loaded once, then borrowed throughout the program. Package lists come from backends, get stored, and are compared to declared packages. The sync plan is computed from these comparisons. Each stage should avoid unnecessary copying.

Design structures to own their essential data. A Package struct should own its name and version as Strings, not borrow them with references. This simplifies lifetime management. Borrowing is for parameter passing and temporary views, not long-term storage.

** TODO Day 12: Ownership Review & Error Exploration

*** Reading

    Rust Error Index: E0382 - use of moved value (https://doc.rust-lang.org/error_codes/E0382.html)
    Rust Error Index: E0502 - conflicting borrows (https://doc.rust-lang.org/error_codes/E0502.html)
    Rust Error Index: E0499 - multiple mutable borrows (https://doc.rust-lang.org/error_codes/E0499.html)
    Rust Error Index: E0505 - move while borrowed (https://doc.rust-lang.org/error_codes/E0505.html)

*** Summary

The Rust compiler produces famously helpful error messages. Today you will intentionally trigger ownership errors and learn to interpret them. This is like a vaccination—controlled exposure builds immunity to confusion later.

E0382 occurs when you use a value after it has been moved. The compiler shows where the move happened and where you tried to use the value afterward. Fixes include cloning before the move, borrowing instead of moving, or restructuring to avoid needing the value after the move.

E0502 occurs when you try to borrow mutably while an immutable borrow is active. The compiler shows both borrow sites. Fixes include ending the immutable borrow earlier, restructuring to separate the operations, or reconsidering whether you need the simultaneous access.

E0499 occurs when you try to create a second mutable borrow while one is active. The fix is similar: end the first borrow or restructure.

E0505 occurs when you try to move a value while it is borrowed. You cannot transfer ownership while someone else has a reference. The fix is ending the borrow before the move.

Reading compiler errors is a skill. Look at the primary error line first, then the notes showing context. Often the note saying "value borrowed here" or "move occurs because" points directly at the issue.

For Nexus development, you will encounter these errors regularly as you build more complex logic. Do not fear them—they are the compiler helping you avoid bugs that would be runtime crashes or data corruption in other languages.

** TODO Day 13: Basic Structs

*** Reading

    The Rust Book, Chapter 5.1: Defining and Instantiating Structs (https://doc.rust-lang.org/book/ch05-01-defining-structs.html)

*** Summary

Structs in Rust are similar to classes in Java, but they contain only data. Methods come separately, which we cover tomorrow. This separation encourages a different design approach where data and behavior are not always bundled together.

A struct definition lists fields with their types. Fields are private by default—external code cannot access them unless you mark them pub. This encapsulation lets you maintain invariants: if a field should only have certain values, you can enforce that by not exposing it directly.

Creating a struct instance requires providing all fields. Rust has no null, so you cannot leave fields uninitialized. If a field might not have a value, you explicitly model that with Option, covered later.

Field init shorthand simplifies construction when your variable names match field names. Instead of writing name: name, you can just write name. This is common in constructors.

Struct update syntax creates a new struct based on an existing one, replacing specific fields. This is useful for making variations without repeating all field values.

Tuple structs have fields without names, accessed by position. They are useful for simple wrappers or when names would not add clarity.

Derive macros automatically implement common traits. Adding #[derive(Debug)] lets you print the struct with {:?}. Adding #[derive(Clone)] lets you call clone(). Adding #[derive(PartialEq)] lets you compare with ==. You will use these constantly.

For Nexus, you need structs for packages, GitHub packages, configuration, sync results, and more. Design them now with all the fields you anticipate needing, but know you can add fields later.

** TODO Day 14: Struct Methods

*** Reading

    The Rust Book, Chapter 5.3: Method Syntax (https://doc.rust-lang.org/book/ch05-03-method-syntax.html)

*** Summary

Methods are functions associated with a struct, defined in impl blocks. The first parameter is some form of self, indicating what the method can do with the struct instance.

Using &self means the method borrows the instance immutably—it can read but not modify. Using &mut self means the method can modify the instance. Using self without reference means the method takes ownership, consuming the instance.

Associated functions without self are like static methods in Java. The most common is a constructor named new that creates and returns an instance. You call these with double colons: Package::new().

The builder pattern works naturally with methods taking self and returning Self. Each method consumes the instance, modifies it, and returns the modified version. Callers chain these for fluent construction.

You can have multiple impl blocks for the same struct. This is useful for organizing code or for conditional compilation. All methods from all impl blocks are available on instances.

Self (capital S) in return position means the struct type itself. This is cleaner than repeating the struct name and works better with generics.

For Nexus, methods make your structs functional. Package should have methods to format itself for display, check if it needs installation, produce the install specification. Config should have methods to validate itself. SyncResult should have methods to record outcomes and produce summaries.

** TODO Day 17: Match Expressions

*** Reading

    The Rust Book, Chapter 6.2: The match Control Flow Construct (https://doc.rust-lang.org/book/ch06-02-match.html)
    Rust By Example: match (https://doc.rust-lang.org/rust-by-example/flow_control/match.html)

*** Summary

Match is Rust's pattern matching construct, far more powerful than switch statements. It lets you compare a value against patterns and execute code based on which pattern matches.

Match is exhaustive—you must handle every possible case. For enums, this means every variant. The compiler enforces this, so adding a variant immediately shows every place that needs to handle it. This is a tremendous maintenance benefit.

Match is an expression returning a value. All arms must return the same type. This makes match useful for computing results based on variants.

Patterns can bind values. When matching an enum variant with data, you extract that data into variables for use in the match arm. This destructuring is concise and powerful.

Multiple patterns can share an arm using the pipe symbol. Ranges match a span of values. Guards add conditions with if after the pattern. The underscore pattern matches anything, serving as a catch-all.

The at-sign binding lets you both test a pattern and bind the whole value. This is useful when you want to check something about a value while also capturing it.

For Nexus, match is central to working with your enums. Handling commands matches Command variants to execute appropriate logic. Processing operation results matches OperationResult variants to format output or collect statistics.

** TODO Day 18: Option<T>

*** Reading

    The Rust Book, Chapter 6.1: Focus on the Option section
    The Rust Book, Chapter 6.3: Concise Control Flow with if let (https://doc.rust-lang.org/book/ch06-03-if-let.html)
    Rust By Example: Option (https://doc.rust-lang.org/rust-by-example/std/option.html)

*** Summary

Rust has no null. Instead, optional values are explicitly typed as Option<T>, an enum with two variants: Some(value) when a value exists, None when it does not.

This design forces explicit handling. You cannot accidentally use a None as if it had a value—the compiler requires you to check. This eliminates null pointer exceptions.

Option has many methods for working with optional values. The is_some and is_none methods check which variant. The unwrap method extracts the value but panics on None—use only when you are certain it is Some. The expect method is like unwrap but with a custom panic message.

Safer extraction methods include unwrap_or, which provides a default if None, and unwrap_or_else, which computes the default lazily with a closure. The question mark operator propagates None early, useful in functions returning Option.

The if let syntax provides concise handling when you only care about the Some case. It is cleaner than a full match when you would just ignore None.

For transformation, map applies a function to the inner value if Some, returning a new Option. The and_then method chains operations that themselves return Option, flattening the result.

For Nexus, Option appears throughout. Package versions might be unspecified. Configuration values might be missing. Search results might not find anything. Using Option makes these cases explicit and forces handling.

** TODO Day 19: Result<T, E> Basics

*** Reading

    The Rust Book, Chapter 9.2: Recoverable Errors with Result (https://doc.rust-lang.org/book/ch09-02-recoverable-errors-with-result.html)

*** Summary

Result is how Rust handles operations that might fail. Like Option, it is an enum: Ok(value) for success, Err(error) for failure. Unlike exceptions in Java, errors are explicit in the return type.

This explicitness is a feature, not a burden. When calling a function that returns Result, you must handle the possibility of failure. The compiler enforces this, preventing forgotten error handling.

Result methods mirror Option methods. The unwrap and expect methods extract Ok values but panic on Err. The unwrap_or and unwrap_or_else methods provide defaults on error. The question mark operator propagates errors, returning early from the function on Err.

Creating Result values is straightforward: wrap success in Ok, failure in Err. Error types can be simple strings, custom error structs, or error enums.

For error type design, consider what callers need to know. Should they distinguish between different failure modes? Then use an enum. Is one error message sufficient? A string or simple struct might suffice. For library code, rich error types help callers respond appropriately.

For Nexus, many operations can fail. Config parsing might find invalid syntax. Backend commands might fail. Network requests might time out. Each should return Result with an error type that helps diagnose problems.

The standard library includes the Error trait for error types. Implementing it enables integration with error handling infrastructure. Your error types should implement Display for human-readable messages and Debug for developer details.

** TODO Day 20: The ? Operator and Error Propagation

*** Reading

    The Rust Book, Chapter 9.2: Focus on the question mark section
    Rust By Example: The question mark operator (https://doc.rust-lang.org/rust-by-example/error/result/question_mark.html)

*** Summary

The question mark operator is syntactic sugar that transforms verbose error handling into concise code. When applied to a Result, it extracts Ok values or returns Err early from the function.

Without question mark, propagating errors requires matching each Result and explicitly returning on Err. With question mark, you write the happy path and let errors propagate automatically.

The operator performs an implicit conversion using the From trait. If your function returns a different error type than the one produced, Rust tries to convert via From. This allows using question mark with different error types if conversions are implemented.

To implement From, you write a function that converts from one type to another. Implementing From for your error type to accept standard library errors enables question mark with those errors.

The question mark operator also works with Option in functions returning Option. It returns None early when encountering None, continuing only on Some.

This operator enables a powerful pattern: functions composed of multiple fallible steps, each marked with question mark, with the happy path reading linearly. Error handling is still rigorous—errors are not ignored—but the code is clean.

For Nexus, loading configuration involves multiple steps that might fail: reading the file, parsing TOML, validating content. Each step can use question mark if you set up proper error conversions.
